ragas_compat.py — what it does

Ragas 0.4.3 imports ChatVertexAI from langchain_community.chat_models.vertexai at module load time. Newer langchain-community (0.4.x) removed that module. The shim registers a stub module so Ragas can import successfully. The platform does not use Vertex AI — it uses OpenAI-compatible judge endpoints via JUDGE_LLM_URL.

# backend/main.py (first import after docstring)
import ragas_compat  # must load before any ragas import

Agent Workflow Tracing — Key Findings

For agent workflows, tracing from middle-step events is easier and richer than black-box endpoint tracing.

Three-tier tracing strategy (from IMPLEMENTATION.md)

TierMechanismStatus
1Poll events → convert to OTEL spans → eval PhoenixImplemented
2Store workflow_trace_id + optional Phoenix deep linkImplemented
3Workflow backend exports OTEL to same Phoenix collectorDeployment config

How events are used

  • backend/targets/agent_studio.py — polls GET /api/workflow/events?trace_id= until kickoff completes or fails
  • backend/trace/workflow_events_to_spans.py — maps events (task, agent, tool, LLM) to OTEL spans for Phoenix
  • backend/trace/event_to_ragas.py — converts events to Ragas messages for agent metrics
  • backend/static/index.html — job detail panel shows per-example event timeline

LLM endpoint vs agent workflow

LLM endpointAgent workflow
VisibilityInput → output onlyTasks, agents, tools, LLM calls
Tracing sourceOpenAI SDK instrumentationWorkflow events API
Tool metricsNot feasible without instrumentationFeasible via event trajectory

langchain_community — What It Is Used For

Not used directly by cai-eval-platform. It appears only as a transitive dependency of Ragas:

  • backend/metrics/ragas_agent.py — Ragas agent metrics (goal accuracy, tool call accuracy/F1)
  • backend/trace/event_to_ragas.py — event → Ragas message conversion

Ragas loads LangChain provider adapters (including Vertex AI) at import time even when the platform only uses OpenAI-compatible judge LLMs.
